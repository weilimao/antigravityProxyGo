### v1.0.6 更新日志

本次更新重点解决了应用在后台运行（如最小化或隐藏到系统托盘）时，由高频网络请求和定时指标推送带来的前端 WebView2 渲染进程内存积压与 CPU 消耗问题，实现了极佳的后台“零开销”静默机制：

1. **窗口状态感知限流（后台零开销）**：
   - 后端 Go 服务引入了线程安全的窗口显示/隐藏状态管理，在托盘显示或启动时标记为可见，在 X 按钮关闭（隐藏到托盘）或最小化时标记为非活动。
   - 综合过滤机制：对高频的请求日志推送、实时指标统计推送以及内存/CPU 状态监测进行了统一的前置拦截限制。当检测到窗口被最小化或彻底隐藏时，后端直接阻断所有 IPC 推送事件。
   - 彻底修复了后台运行高频代理请求时，前端渲染进程持续接收日志、反序列化大 JSON 以及修改 DOM 树而引起的渲染器进程内存持续上涨的问题。

2. **周期性数据包推送深度限制**：
   - 对内存监控中每 60 秒触发一次的周期性 `stats-updated` 广播进行了过滤，确保在隐藏/最小化状态下，不向前端传送包含 720 个 Trends 趋势点的庞大 Payload，彻底切断了后台的数据流，换取极佳的 CPU 和内存占用稳定性。

3. **前端按需还原同步机制**：
   - 前端添加了对 visibilitychange 的事件监听，在后台运行时完全保持静置（使得 `msedgewebview2` 渲染进程在最小化下缩减至 ~77MB，隐藏下缩减至 ~94MB，CPU 彻底归零）。
   - 在用户通过双击托盘重新激活还原窗口的瞬间，前端会通过一次性的 `get-state` 同步拉取后台期间漏掉的最新 50 条日志与指标卡片数据并重绘，做到了内存优化与界面状态同步的完美共存。
